# Internationalisation (i18n) – Application bilingue

**Dernière mise à jour :** 23 février 2026 (calendar.tile.athleteCommentLabel)  
**Langues supportées :** Français (fr) – défaut, Anglais (en)  
**Bibliothèque :** next-intl 3.x  
**Statut :** ✅ Mise en place bilingue complétée

---

## 🎯 Règle d'or : Toujours penser bilingue

**Toute nouvelle feature ou modification de texte visible par l'utilisateur doit être traduite dès le départ.**

- Ne pas introduire de chaînes en dur (français ou anglais) dans les composants ou les server actions.
- Ajouter les clés dans `messages/fr.json` et `messages/en.json` en même temps que le code.
- Utiliser les namespaces existants ou en créer un si le domaine le justifie.

Consulter la **Checklist nouvelles features** plus bas pour ne rien oublier.

---

## 📋 Résumé de l'implémentation

- **Infrastructure :** next-intl, middleware (intégré dans `proxy.ts`), structure `i18n/` et `messages/`
- **Routing :** Routes sous `app/[locale]/` ; API et auth restent à la racine
- **Fichiers de traduction :** `messages/fr.json`, `messages/en.json` (namespaces : common, auth, navigation, sports, workouts, goals, profile, coach, devices, chat, errors, metadata, calendar, etc. — ex. `calendar.today` pour le badge « Aujourd'hui »)
- **Composant :** `LanguageSwitcher` sur la page profil (et page d’accueil) pour basculer FR/EN, avec enregistrement de `preferred_locale` en BD

---

## 🗂️ Structure des fichiers

```
/i18n
  ├── request.ts      # Configuration i18n
  └── routing.ts      # Locales (fr, en), defaultLocale, localePrefix
/messages
  ├── fr.json         # Traductions françaises
  └── en.json         # Traductions anglaises
/app
  ├── [locale]/       # Routes localisées (page, dashboard, login, etc.)
  ├── api/            # API routes (non localisées)
  └── auth/           # Auth callbacks (non localisées)
proxy.ts              # Middleware : next-intl + Supabase auth
```

## 🎯 Utilisation dans le code

### Server Component

```typescript
import { useTranslations } from 'next-intl';

export default function MyPage() {
  const t = useTranslations('navigation');
  return <h1>{t('dashboard')}</h1>;
}
```

### Client Component

```typescript
'use client';
import { useTranslations } from 'next-intl';

export default function MyComponent() {
  const t = useTranslations('common');
  return <button>{t('save')}</button>;
}
```

### Server Actions (messages d'erreur / succès)

```typescript
'use server';
import { getTranslations } from 'next-intl/server';

export async function myAction(formData: FormData) {
  const locale = (formData.get('_locale') as string) || 'fr';
  const t = await getTranslations({ locale, namespace: 'auth.errors' });
  if (!user) return createError(t('notAuthenticated'), 'AUTH_REQUIRED');
  return createSuccess(data);
}
```

Transmettre la locale depuis le formulaire : `formData.append('_locale', locale)` (avec `useLocale()` côté client).

### Labels de sports dynamiques

import { SPORT_TRANSLATION_KEYS } from '@/lib/sportStyles';
import { useTranslations } from 'next-intl';

const t = useTranslations();

## 🌐 URLs et routing

- **Locale par défaut :** Français (fr) – pas de préfixe : `/dashboard`
- **Anglais :** Préfixe `/en` : `/en/dashboard`
- **Middleware :** Détection de la langue du navigateur, redirections avec conservation de la locale (proxy combine next-intl + Supabase auth).

## ✅ Checklist : Nouvelles features (toujours penser bilingue)

À chaque nouvelle feature ou modification de texte :

1. **Composants (Server ou Client)**
   - [ ] Aucun texte en dur : utiliser `useTranslations('namespace')` et `t('key')`.
   - [ ] Clés ajoutées dans `messages/fr.json` et `messages/en.json` (même structure).

2. **Formulaires et Server Actions**
   - [ ] Passer la locale aux actions : `formData.append('_locale', locale)` (récupérer `locale` avec `useLocale()`).
   - [ ] Messages d'erreur / succès traduits via `getTranslations({ locale, namespace })` dans l’action.

3. **Pages**
   - [ ] Titres et meta : utiliser les clés du namespace `metadata` (ou équivalent) avec la locale.

4. **Nouveau domaine (optionnel)**
   - [ ] Si besoin, créer un namespace dédié dans `fr.json` et `en.json` (ex. `myFeature`) et l’utiliser de façon cohérente.

5. **Vérifications rapides**
   - [ ] Les deux fichiers JSON ont les mêmes clés pour le namespace utilisé.
   - [ ] Tester le flux en FR et en EN (changement de langue via `LanguageSwitcher`).

## ➕ Ajouter une nouvelle langue

1. Créer `messages/xx.json` (copier depuis `en.json` et traduire).
2. Mettre à jour `i18n/routing.ts` : ajouter la locale dans `locales`.
3. Mettre à jour le matcher dans `proxy.ts` si nécessaire.
4. Ajouter l’option dans le composant `LanguageSwitcher`.

---

## 🐛 Dépannage

- **Traductions manquantes :** Vérifier que la clé existe dans les deux fichiers et que le namespace est correct (`t('namespace.key')` ou `useTranslations('namespace')` puis `t('key')`).
- **Routing :** Les routes utilisateur sont sous `app/[locale]/` ; ne pas mettre les routes API/auth dans `[locale]`.
- **Params Next.js 16 :** Dans les layouts/pages, `params` est une Promise : `const { locale } = await params;`.
